## 架构概述

### 整体架构

Agent 模块采用**事件驱动架构**和**循环执行模式**，通过以下层次实现：

```
┌─────────────────────────────────────────┐
│         Agent (代理主类)                 │
│  ┌───────────────────────────────────┐  │
│  │        执行循环 (run)              │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │  1. 消息摘要检查            │  │  │
│  │  └─────────────────────────────┘  │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │  2. LLM 生成响应            │  │  │
│  │  └─────────────────────────────┘  │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │  3. 处理工具调用             │  │  │
│  │  └─────────────────────────────┘  │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │  4. 执行工具                 │  │  │
│  │  └─────────────────────────────┘  │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │  5. 更新消息历史             │  │  │
│  │  └─────────────────────────────┘  │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │        消息管理层                 │  │
│  │  - 消息历史维护                   │  │
│  │  - Token 估算                     │  │
│  │  - 智能摘要                       │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │        输出管理层                 │  │
│  │  - 彩色终端输出                   │  │
│  │  - 思考内容显示                   │  │
│  │  - 工具调用显示                   │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
           │                │
           ▼                ▼
┌─────────────────┐  ┌─────────────────┐
│   LLM 客户端    │  │    工具集       │
│                 │  │                 │
│ - OpenAI Client │  │ - Base Tool     │
│ - 其他提供商    │  │ - Tool Result   │
│                 │  │ - 自定义工具    │
└─────────────────┘  └─────────────────┘
```

### 架构特点

1. **事件驱动**：每个执行步骤都是事件驱动的
2. **循环执行**：通过 `run()` 方法实现循环执行模式
3. **分层设计**：消息管理层、LLM 交互层、工具调用层、输出层
4. **异步优先**：所有操作都是异步的，充分利用 asyncio
5. **可扩展**：通过工具系统可以轻松扩展功能


## 设计模式

### 1. 工厂模式（Factory Pattern）

**应用**：通过 `LLMClient` 工厂类创建不同的 LLM 客户端

```python
class LLMClient:
    @staticmethod
    def create(provider, **kwargs) -> LLMClientBase:
        if provider == LLMProvider.OPENAI:
            return OpenAIClient(**kwargs)
        # 可扩展其他提供商
```

### 2. 策略模式（Strategy Pattern）

**应用**：不同的 Token 估算策略

```python
def _estimate_tokens(self) -> int:
    try:
        # 策略 1：使用 tiktoken（精确）
        encoding = tiktoken.get_encoding("cl100k_base")
        return self._estimate_with_encoding(encoding)
    except Exception:
        # 策略 2：回退到字符估算（快速）
        return self._estimate_tokens_fallback()
```

### 3. 迭代器模式（Iterator Pattern）

**应用**：消息历史的迭代和处理

```python
for msg in self.messages:
    # 处理每条消息
    process_message(msg)
```

### 4. 模板方法模式（Template Method Pattern）

**应用**：`run()` 方法定义执行骨架，子类可重写特定步骤

```python
async def run(self) -> str:
    # 模板方法定义了执行骨架
    while step < self.max_steps:
        await self._summarize_messages()  # 钩子方法
        response = await self._generate_response()  # 抽象方法
        await self._process_response(response)  # 钩子方法
```

### 5. 装饰器模式（Decorator Pattern）

**应用**：通过 `Colors` 装饰文本输出

```python
def print_colored(text: str, color: str):
    print(f"{color}{text}{Colors.RESET}")
```

## 执行流程

### 完整执行流程图

```mermaid
sequenceDiagram
    participant U as User
    participant A as Agent
    participant L as LLM Client
    participant T as Tools
    participant M as Message History

    U->>A: add_user_message()
    A->>M: 添加用户消息

    loop 执行步数 < max_steps
        A->>A: _summarize_messages()
        Note over A: 检查 Token 限制
        alt Token 超过限制
            A->>A: _create_summary()
            A->>M: 替换为摘要消息
        end

        A->>L: generate(messages, tools)
        L-->>A: LLMResponse

        A->>A: 显示思考内容
        A->>A: 显示响应内容

        alt 有工具调用
            loop 每个工具调用
                A->>T: execute(**params)
                T-->>A: ToolResult
                A->>A: 显示工具结果
                A->>M: 添加 tool 消息
            end
        end

        A->>M: 添加 assistant 消息

        alt 无工具调用且任务完成
            A-->>U: 返回结果
            break
        end
    end

    Note over A: 达到最大步数
    A-->>U: 返回错误消息
```

### 步骤详解

#### 步骤 1：消息摘要检查

```python
async def _summarize_messages(self):
    estimated_tokens = self._estimate_tokens()

    # 如果没有超过限制，直接返回
    if estimated_tokens <= self.token_limit:
        return

    print(f"Token 估计: {estimated_tokens}/{self.token_limit}")
    print("触发消息摘要...")

    # 执行摘要逻辑
    new_messages = [self.messages[0]]  # 保留系统消息
    # ... 摘要处理逻辑

    self.messages = new_messages
```

**关键特性**：
- 保留所有用户消息（用户意图）
- 摘要每个执行轮次的内容
- 结构化消息：system → user1 → summary1 → user2 → summary2

#### 步骤 2：LLM 生成响应

```python
response = await self.llm.generate(
    messages=self.messages,
    tools=list(self.tools.values())
)
```

**处理流程**：
1. 准备消息和工具
2. 异步调用 LLM API
3. 接收 LLMResponse
4. 验证响应格式

#### 步骤 3：显示思考内容

```python
if response.thinking:
    print(f"\n{Colors.BOLD}{Colors.MAGENTA}思考:{Colors.RESET}")
    print(f"{Colors.DIM}{response.thinking}{Colors.RESET}")
```

#### 步骤 4：处理工具调用

```python
for tool_call in response.tool_calls:
    function_name = tool_call.function.name
    arguments = tool_call.function.arguments

    # 执行工具
    if function_name in self.tools:
        tool = self.tools[function_name]
        result = await tool.execute(**arguments)

    # 添加结果消息
    tool_msg = Message(
        role="tool",
        content=result.content if result.success else f"错误: {result.error}",
        tool_call_id=tool_call_id,
        name=function_name
    )
    self.messages.append(tool_msg)
```

## 消息管理机制

### 消息类型

| 类型 | 说明 | 包含字段 |
|------|------|----------|
| system | 系统消息，设置 AI 角色和行为 | role, content |
| user | 用户消息，用户输入 | role, content |
| assistant | AI 响应，可能包含思考和工具调用 | role, content, thinking, tool_calls |
| tool | 工具执行结果 | role, content, tool_call_id, name |

### 消息历史结构

**初始化后**：
```
[
    Message(role="system", content="系统提示")
]
```

**添加用户消息**：
```
[
    Message(role="system", content="系统提示"),
    Message(role="user", content="用户问题")
]
```

**LLM 响应后**：
```
[
    Message(role="system", content="系统提示"),
    Message(role="user", content="用户问题"),
    Message(role="assistant", content="AI 回答", thinking="思考过程")
]
```

**工具调用后**：
```
[
    Message(role="system", content="系统提示"),
    Message(role="user", content="用户问题"),
    Message(role="assistant", content="我将使用工具", tool_calls=[...]),
    Message(role="tool", content="工具结果", tool_call_id="...")
]
```

### 消息摘要策略

**目标**：保持关键信息，减少 Token 使用

**策略**：
1. **保留所有用户消息**：用户意图不能丢失
2. **摘要执行过程**：压缩 assistant 和 tool 消息
3. **保留最后对话**：保留最近的完整对话
4. **结构化摘要**：

```
[System] + [User1] + [Summary1] + [User2] + [Summary2] + [User3] + [Executing]
```

**摘要生成流程**：

```python
async def _create_summary(self, messages: list[Message], round_num: int) -> str:
    # 提取执行过程
    summary_content = f"Round {round_num} 执行过程:\n\n"
    for msg in messages:
        if msg.role == "assistant":
            summary_content += f"Assistant: {msg.content}\n"
            if msg.tool_calls:
                tool_names = [tc.function.name for tc in msg.tool_calls]
                summary_content += f" -> 调用工具: {', '.join(tool_names)}\n"
        elif msg.role == "tool":
            summary_content += f" <- 工具返回: {msg.content}\n"

    # 调用 LLM 生成摘要
    summary_prompt = f"请提供以下执行过程的简洁摘要..."
    summary_response = await self.llm.generate(...)
    return summary_response.content
```

## 工具调用机制

### 工具注册

```python
self.tools = {tool.name: tool for tool in tools}
```

### 工具执行流程

```python
async def _execute_tool(self, tool_call):
    function_name = tool_call.function.name
    arguments = tool_call.function.arguments

    # 检查工具是否存在
    if function_name not in self.tools:
        return ToolResult(
            success=False,
            content="",
            error=f"未知工具: {function_name}"
        )

    # 执行工具
    try:
        tool = self.tools[function_name]
        result = await tool.execute(**arguments)
        return result
    except Exception as e:
        return ToolResult(
            success=False,
            content="",
            error=f"工具执行失败: {type(e).__name__}: {str(e)}"
        )
```

**1. 单一职责**：每个工具只做一件事

```python
class FileReaderTool(Tool):

    name = "read_file"
    description = "读取文件内容"

    async def execute(self, file_path: str) -> ToolResult:
        # 只实现文件读取逻辑
        pass
```

**2. 清晰的参数**：

```python
async def execute(
    self,
    file_path: str,           # 必需参数
    encoding: str = "utf-8",  # 可选参数
    max_lines: int = 1000     # 可选参数
) -> ToolResult:
```

**3. 完善的错误处理**：

```python
async def execute(self, file_path: str) -> ToolResult:
    if not os.path.exists(file_path):
        return ToolResult(
            success=False,
            content="",
            error=f"文件不存在: {file_path}"
        )

    try:
        with open(file_path, 'r') as f:
            content = f.read()
        return ToolResult(success=True, content=content)
    except Exception as e:
        return ToolResult(
            success=False,
            content="",
            error=f"读取失败: {str(e)}"
        )
```

## Token 管理与摘要

### Token 估算算法

**方法 1：精确估算（tiktoken）**

```python
def _estimate_tokens(self) -> int:
    encoding = tiktoken.get_encoding("cl100k_base")
    total_tokens = 0

    for msg in self.messages:
        # 计算文本内容
        if isinstance(msg.content, str):
            total_tokens += len(encoding.encode(msg.content))

        # 计算思考内容
        if msg.thinking:
            total_tokens += len(encoding.encode(msg.thinking))

        # 计算工具调用
        if msg.tool_calls:
            total_tokens += len(encoding.encode(str(msg.tool_calls)))

        # 消息元数据开销（约 4 tokens）
        total_tokens += 4

    return total_tokens
```

**方法 2：快速估算（回退）**

```python
def _estimate_tokens_fallback(self) -> int:
    total_tokens = 0

    for msg in self.messages:
        if isinstance(msg.content, str):
            total_tokens += len(msg.content)

        if msg.thinking:
            total_tokens += len(msg.thinking)

        if msg.tool_calls:
            total_tokens += len(str(msg.tool_calls))

    # 平均 2.5 字符 per token
    return int(total_tokens / 2.5)
```

### 摘要触发条件

```python
async def run(self):
    while step < self.max_steps:
        # 每次循环都检查 Token 限制
        await self._summarize_messages()

        # ... 继续执行
```

**触发条件**：
- `estimated_tokens > token_limit`
- 默认 token_limit = 80000

### 摘要优化策略

**1. 早期检查**：

```python
# 在生成响应前检查
await self._summarize_messages()
response = await self.llm.generate(self.messages, tools)
```

**2. 最小化摘要频率**：

```python
# 只有在真正超过限制时才摘要
if estimated_tokens <= self.token_limit:
    return  # 提前返回，不执行摘要
```

**3. 结构化摘要**：

```
原始消息: [System] [U1] [A1] [T1] [A2] [T2] [A3] [U2] [A4]
          ↓
摘要后:    [System] [U1] [Sum1] [U2] [Executing]
```

## 错误处理机制

### 错误类型

**1. LLM 调用失败**

```python
try:
    response = await self.llm.generate(messages, tools)
except Exception as e:
    error_msg = f"LLM 调用失败: {str(e)}"
    print(f"\n{Colors.BRIGHT_RED}❌ 错误:{Colors.RESET} {error_msg}")
    return error_msg
```

**2. 工具不存在**

```python
if function_name not in self.tools:
    result = ToolResult(
        success=False,
        content="",
        error=f"未知工具: {function_name}"
    )
```

**3. 工具执行失败**

```python
try:
    result = await tool.execute(**arguments)
except Exception as e:
    import traceback

    error_detail = f"{type(e).__name__}: {str(e)}"
    error_trace = traceback.format_exc()

    result = ToolResult(
        success=False,
        content="",
        error=f"工具执行失败: {error_detail}\n\nTraceback:\n{error_trace}"
    )
```

### 错误恢复

**策略 1：继续执行**

```python
# 工具失败后，继续执行下一轮
# LLM 可以基于错误信息进行调整
self.messages.append(tool_msg)  # 添加错误消息
step += 1
```

**策略 2：提前返回**

```python
# LLM 调用失败，返回错误信息
if not response.tool_calls:
    return response.content  # 正常完成
```

**策略 3：超时保护**


## 设计决策

### 决策 1：为什么使用异步？

**选择**：全面采用 async/await

**原因**：
1. **性能**：避免阻塞 I/O 操作
2. **并发**：可同时执行多个 Agent
3. **响应性**：更好的用户体验

**替代方案**：
- 同步执行：简单但性能差
- 多线程：复杂且有 GIL 限制

### 决策 2：为什么需要消息摘要？

**选择**：智能消息摘要机制

**原因**：
1. **Token 限制**：LLM 有上下文长度限制
2. **成本控制**：减少 Token 使用
3. **性能优化**：减少响应时间

**替代方案**：
- 不摘要：可能因 Token 过多失败
- 固定窗口：丢失重要上下文
- 滑窗：丢失早期关键信息

### 决策 3：为什么使用颜色输出？

**选择**：ANSI 彩色终端输出

**原因**：
1. **可读性**：不同类型信息用不同颜色区分
2. **用户体验**：清晰的视觉层次
3. **调试友好**：快速定位问题

**替代方案**：
- 普通文本：信息层次不清晰
- HTML 输出：仅适用于 Web 界面

### 决策 4：为什么工具执行失败不抛出异常？

**选择**：转换为 ToolResult 对象

**原因**：
1. **优雅处理**：工具失败是正常情况
2. **错误传播**：错误信息可传递给 LLM
3. **执行继续**：单个工具失败不影响整体

**替代方案**：
- 抛出异常：中断执行，可能丢失其他工具调用机会
- 返回 None：无法区分成功和失败

### 决策 5：为什么需要 max_steps？

**选择**：设置最大执行步数

**原因**：
1. **防止无限循环**：避免无休止的工具调用
2. **成本控制**：限制 Token 消耗
3. **超时保护**：确保任务在合理时间内完成

**替代方案**：
- 无限制：可能导致无限循环
- 基于时间：不如基于步骤精确

### 决策 6：为什么保留完整的消息历史？

**选择**：在摘要前保留所有消息

**原因**：
1. **准确性**：用户意图不会丢失
2. **可追溯**：可以查看完整的执行过程
3. **调试友好**：完整的历史有助于问题定位

**替代方案**：
- 立即摘要：可能丢失关键信息
- 完全不摘要：Token 限制问题

### 决策 7：为什么使用 tiktoken？

**选择**：使用 tiktoken 进行精确 Token 计数

**原因**：
1. **准确性**：与 LLM 实际计数一致
2. **可靠性**：OpenAI 官方库
3. **性能**：编码效率高

**替代方案**：
- 字符估算：不准确
- 第三方库：可能不兼容
- 人工估算：不可行

### 决策 8：为什么工作空间用 pathlib？

**选择**：使用 pathlib.Path

**原因**：
1. **跨平台**：自动处理不同操作系统的路径差异
2. **面向对象**：提供丰富的方法
3. **类型安全**：Path 对象提供类型检查

**替代方案**：
- 字符串路径：需要手动处理路径分隔符
- os.path：较为繁琐的 API
